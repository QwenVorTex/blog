## 2. git

a. git的使用是必要的，很多操作记得遵循便于回溯历史提交，使用git push时，有时候会报错，这是因为你本地的git仓库和远程仓库的两个分支没有建立联系（对了说到这，安装git时建议把默认分支从master改为main，方便很多），使用git push origin <branch>即可。

b. 还要注意保持溯源的简便性，当你与他人共同合作时，方便的操作时两人各自在远程库的不同分支上各自修改，最后merge，在极端情况下一定要链接同一个分支，也请谨慎操作，你本地库不是最新版本时git push是推送不出去的，一定要先git pull一下远程库，解释一下，git pull是git fetch和git merge的组合，会在提交信息里添加一次无用的merge信息，影响搜索commit历史。为了解决这个问题，请使用git pull --rebase，保证线性整洁。

c. 查看commit历史使用可以使用git log --pretty=oneline，或者直接git log --oneline，后者的提交哈希值会缩略，这两个都在一行内显示每次commit，易读性比较好

d. 善用git版本回退，共有两种操作，git reset与git revert，我们先来讲第一种。

1.  git reset

    如同在C语言中学习的node list一样，git内置一个HEAD指针，负责指向当前的节点，这个节点的数据也就是当前版本的内容。

    打个比方，当前有四个版本，分别是A，B，C，D，A和B是正确提交的版本，C和D则是错误的内容，此时使用git reset --hard <hash>则相当于是将HEAD指针指向了B。后面的节点还在啊，但是B和后面节点的链接已经断了，自然不需要再考虑。之后继续按照git add <>，git commit -m ""操作即可。但是本地库的HEAD指向是B了，此时远程库的HEAD可还指向D呢，直接git push是会报错的，因为本地版本落后于远程库版本，此时就要使用git push -f强制推送。

2.  git revert
    1. git revert <hash>的功能请理解为在D的后面增加了一个新的节点E，只不过E的数据和C一模一样，记住：revert操作需要一个一个来，你想要回退到D，就要先使用D的哈希值执行操作，再用C的哈希值执行操作。
    2. 批量回退的形式如git revert C^..D，这里的含义为回退C到D的所有版本，具体操作时请使用对应哈希值代替节点名。批量回退默认将每一个版本回退单独提交，也就是版本提交信息和上面一个一个回退一样，如果你希望把所有回退统一到一个节点，可以使用git revert -n C^..D，仅需要添加一个-n参数。

    不难看出，revert操作比reset多了一个回退版本的提交历史，这样你半夜突然想起来那个错误版本中有一个绝妙的点子时就不用悔得恨不得杀了自己。

    再对版本信息做一个补充，提交信息是可以合并的，所以不用担心push了一堆史上去，用rebase命令可以对想要合并的节点进行合并，这是一个图文化的操作，不做描述，看视频很好懂，用文字描述实在太看得起我的语文水平了。
