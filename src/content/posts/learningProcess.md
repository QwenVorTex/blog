## 1. 数据库

对于数据库的选择，我更推荐 PostgreSQL，不过如果是面向任务编程，MySQL 也完全够用。这里我要说一句比较主观的话：很多项目里虽然用了 MySQL，但最后其实也只是把它当成一个更强一点的表格系统，外键之类的功能压根没怎么碰。说回来，任务驱动的前提下，能把事情做完的数据库就是好数据库，MySQL 好上手，这点就是它很大的优势。

## 2. git

a. git 的使用是必要的，很多操作都要以“便于回溯历史提交”为前提。使用 `git push` 时有时候会报错，这是因为你本地分支和远程分支的对应关系还没建好（对了，说到这，安装 git 时建议把默认分支从 `master` 改成 `main`，方便很多），这时候使用 `git push origin <branch>` 即可。

b. 还要注意保持溯源的简便性。当你与他人共同合作时，比较方便的操作是两个人在远程库的不同分支上各自修改，最后再 merge。极端情况下如果一定要共用同一个分支，也请谨慎操作。你本地库不是最新版本时，`git push` 是推送不出去的，一定要先拉一下远程库。这里解释一下，`git pull` 本质上是 `git fetch` 和 `git merge` 的组合，会在提交信息里加一次没那么好看的 merge 记录，影响搜索 commit 历史。为了解决这个问题，请使用 `git pull --rebase`，保证历史线性整洁。

c. 查看 commit 历史时可以使用 `git log --pretty=oneline`，或者直接 `git log --oneline`，后者的提交哈希值会缩略。这两个都会在一行内显示每次 commit，易读性比较好。

d. 善用 git 版本回退，共有两种操作，`git reset` 与 `git revert`，我们先来讲第一种。

1.  git reset

    如同在 C 语言里学过的 node list 一样，git 内置一个 `HEAD` 指针，负责指向当前节点，这个节点的数据也就是当前版本的内容。

    打个比方，当前有四个版本，分别是 A、B、C、D，A 和 B 是正确提交的版本，C 和 D 则是错误内容。此时使用 `git reset --hard <hash>`，相当于是把 `HEAD` 指针指回了 B。后面的节点还在，但 B 和后面节点的链接已经断了，自然不需要再考虑。之后继续按照 `git add <>`、`git commit -m ""` 操作即可。但是本地库的 `HEAD` 指向已经是 B 了，此时远程库的 `HEAD` 可能还指向 D，直接 `git push` 会报错，因为本地版本落后于远程库版本，这时候就要使用 `git push -f` 强制推送。

2.  git revert
    1. `git revert <hash>` 的功能可以理解为在 D 的后面增加了一个新的节点 E，只不过 E 的数据和“撤销后的结果”一致。记住：`revert` 操作通常是一个一个来。你如果想把 C 和 D 这两次提交都撤掉，就要先用 D 的哈希值执行一次，再用 C 的哈希值执行一次。
    2. 批量回退的形式如 `git revert C^..D`，这里的含义是回退 C 到 D 的所有版本，具体操作时请使用对应哈希值代替节点名。批量回退默认会把每一个版本的回退都单独提交，也就是提交信息和上面一个个回退一样。如果你希望把所有回退统一到一个节点，可以使用 `git revert -n C^..D`，只需要多加一个 `-n` 参数。

    不难看出，`revert` 操作比 `reset` 多保留了一段回退历史，这样你半夜突然想起来那个错误版本里其实有个绝妙点子时，就不用悔得想砸键盘。

    再对版本信息做一个补充，提交信息是可以合并的，所以不用担心 push 了一堆历史上去。用 `rebase` 命令可以把想合并的节点合并掉，这是一个比较图形化的操作，不展开描述了，看视频很好懂，用文字描述实在太看得起我的语文水平了。

## 3. 语言

1. JavaScript 虽然简单，但是非常有用，和它的超集 TypeScript（请忽略华为超集的超集 ArkTS）结合可以完成很多事，JS + AWS 的组合也可以很快完成部署。同样反直觉的是，我以前看过一些 benchmark，JS 在脚本语言里的性能其实并不差，至少远没有很多人想象中那么离谱。

2. C# 从很多体验上看我确实更喜欢，不过就业环境里 Java 还是更常见一点。只是这几年整个生态也在变，不少团队已经开始往 Go 和 Rust 那边走了。在 .NET 平台上运行 C# 有种数值上的美（巨硬钞能力），不过对于后端方向，没必要强制规定自己一定学哪一种语言，能够完成任务即可。快去学计算机领域的原神Rust。

3. Android 开发里 Kotlin 我是更推荐的，不过目前市面上的学习资料连 Java 开发 Android 都不算多，Kotlin 就更少了。但是如果愿意硬啃 Kotlin，我觉得是值得的。
